eat-thing Architecture Audit Recommendations

Recommended direction for each difference between the current implementation and ARCHITECTURE.md.

Difference Better approach Reason
Household scoping missing from some shopping-list price/cart routes Docs are right. Fix code. This is a security boundary. Every user-facing query and mutation must verify household_id. Add household_id to shopping_list_prices if strict row-level tenancy is the intended invariant.
Inventory described as event-derived, but CRUD edits rows directly Code is more practical for now. Update docs. Fully event-sourced inventory requires events for purchases, corrections, deletions, expiry, and reconciliation. Current cook_events alone cannot derive inventory. Document inventory as mutable balance rows with append-only cooking audit events. Consider a full inventory ledger later only if history becomes valuable.
Canonical foods described as curated, but manual actions silently insert them Docs are right long-term. Fix code deliberately. Silent global taxonomy growth will accumulate duplicates and inconsistent categories. A lightweight "confirm new canonical food" prompt is enough initially. Until implemented, docs should say review is planned rather than enforced.
Shopping-list generation claims staples below threshold are included, but /from-plan only derives recipe items Docs describe the better product behavior. Fix code. Low-stock staples are core shopping-list behavior and already part of the schema. Implement one shared low-stock derivation used by /from-plan and the inventory sidebar.
Scraper described as launchd-supervised, but scraper plist is backlog work Target architecture is good, but docs should state current status accurately. Keep launchd as the intended deployment model. Change docs to "planned" until Slice 2 lands. Delete the obsolete OpenBrain plist now.
Docs say photo rows store storage paths and signed URLs; code stores public recipe URLs and has no inventory photos Choose based on privacy needs. Recommended: keep code for now and update docs. Public recipe images are operationally simpler and low-risk for a two-user household. If imported recipe photos may contain personal content, switch the bucket to private and store paths. Inventory photos should be documented as future work or removed from the architecture claim.
Docs say all stored units are canonical; code preserves recipe text units and allows arbitrary inventory units Use a hybrid model. Code needs tightening and docs need clarification. Preserving original recipe units is correct for display. Inventory quantities should be restricted to canonical g, ml, or count. Recipe ingredients should store original text plus normalized metric metadata for calculations.
Docs imply active-household selection; middleware picks the first membership Code is adequate only while one household exists. Docs should say that explicitly. Do not build a household switcher yet. Add deterministic selection or an active-household mechanism before allowing a second household.
Topology places Meal Planner import on the Mac mini; deployed server can call HTTP MCP directly Code is better. Update docs. HTTP MCP from the API is simpler. Keep local stdio as a development fallback. Meal Planner import should appear beside apps/server, not inside the Mac-mini worker box.
Server reads SCRAPER_HMAC_SECRET; worker reads WORKER_HMAC_KEY Fix code and standardize on one name. Use SCRAPER_HMAC_SECRET everywhere because it is specific and already used by the server tests. Update deployment templates and docs.
Docs say every domain table carries household_id; canonical_foods and shopping_list_prices do not Split the cases. canonical_foods should remain global and curated. shopping_list_prices should gain household_id if strict tenancy is the rule. Docs should explicitly list global reference tables and Better-Auth-owned tables as exceptions.
Docs call the auth table users; Better-Auth owns user, session, account, and verification Code is authoritative. Update docs. Document the actual Better-Auth schema.
Docs say a date holds at most four recipes; server permits more Recommended: fix code. If four is a product rule, enforce it server-side so alternate clients cannot violate it. If it is only a layout preference, change docs to say the UI caps visible additions at four.
recipes.total_time_minutes and tags exist, but recipe write routes cannot populate them Schema and UI direction are better. Fix code. Accept and persist these fields in recipe create/update routes, especially for imported recipes.
Obsolete OpenBrain launchd plist remains after D22 Delete the file. It references removed code and can mislead deployment work.
Shared scraper job type omits shipped add_to_cart Fix code. Add add_to_cart to the shared union so the shared contract matches the implemented API.
Docs describe list finalization, but no finalize behavior exists Update docs unless finalization is needed soon. finalized_at is currently unused. Describe the current list as editable and remove "before finalizing," or add finalization to the roadmap.
Docs describe prompted responses as refining future deductions Update docs. Responses are recorded but no learning mechanism consumes them yet. Say they are retained for audit and future refinement.

Suggested Priority

  1. Fix tenancy checks and HMAC naming.
  2. Implement low-stock staples.
  3. Align shared types and recipe write fields.
  4. Remove the stale plist.
  5. Revise ARCHITECTURE.md to describe mutable inventory, unit handling, Meal Planner topology, current launchd status, photo storage, and planned taxonomy enforcement accurately.
  6. Decide whether the four-recipe cap and taxonomy review should become server invariants immediately.
